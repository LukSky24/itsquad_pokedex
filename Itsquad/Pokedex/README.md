# Itsquad_Pokedex Module

## Overview

A Magento 2 module that provides a frontend Pokedex search interface. Users can search for Pokemon by ID and view detailed information fetched from the [PokeAPI](https://pokeapi.co/).

## Features

- **Search Pokemon by ID** - Simple form to search Pokemon (1-1000+)
- **Display Pokemon Details** - Shows ID, name, types, height, weight, base experience, and sprite image
- **Query Logging** - All API queries are logged to `var/log/pokemon.log`
- **Public Frontend Page** - Available at `/pokedex` route

## Design Patterns Used

This module demonstrates clean architecture principles and multiple design patterns:

### 1. **Repository Pattern**
- **Files**: `Api/PokemonRepositoryInterface.php`, `Model/PokemonRepository.php`
- **Purpose**: Abstracts data access layer from business logic
- **Benefits**: Easy to swap data sources (API, database, cache), testable

### 2. **Strategy Pattern**
- **Files**: `Api/Service/PokemonApiInterface.php`, `Service/PokeApiService.php`
- **Purpose**: Defines a family of interchangeable algorithms (API implementations)
- **Benefits**: Open/Closed Principle - can add new API sources without modifying existing code

### 2a. **Adapter Pattern**
- **Files**: `Api/Client/PokeApiClientInterface.php`, `Client/GuzzlePokeApiClient.php`
- **Purpose**: Wraps the external HTTP client (GuzzleHTTP) behind an interface, adapting it to the module's needs
- **Benefits**: Swappable HTTP clients (e.g. replace Guzzle with cURL or Symfony HttpClient), Single Responsibility Principle (service handles data transformation, client handles HTTP), easier unit testing of each layer independently

### 3. **Data Transfer Object (DTO)**
- **Files**: `Api/Data/PokemonInterface.php`, `Model/Data/Pokemon.php`
- **Purpose**: Type-safe data containers with explicit getters/setters and field name constants
- **Benefits**: Better IDE autocomplete, type safety, validation, consistent field name references via constants

### 4. **Factory Pattern**
- **File**: `Model/Data/PokemonFactory.php` (auto-generated by Magento)
- **Purpose**: Encapsulates object creation
- **Benefits**: Centralized object instantiation, easier testing

### 5. **Registry Pattern (In-Memory Cache)**
- **File**: `Model/PokemonRegistry.php`
- **Purpose**: In-memory cache for Pokemon objects retrieved from the API during a single request
- **Methods**:
  - `retrieve(int $objectId)` - Returns cached Pokemon or fetches from API, hydrates DTO, caches via `push()`, and returns
  - `push(PokemonInterface $object)` - Manually adds a Pokemon to the cache
- **Benefits**: Avoids duplicate API calls for the same Pokemon within a request lifecycle

### 6. **Plugin Pattern (AOP)**
- **File**: `Plugin/PokemonApiLoggerPlugin.php`
- **Purpose**: Aspect-Oriented Programming - add behavior without modifying source code
- **Benefits**: Logging, caching, or modifying behavior of third-party code

### 7. **Dependency Injection**
- **File**: `etc/di.xml`
- **Purpose**: Inversion of Control container configuration
- **Benefits**: Loose coupling, easier testing, configuration-based wiring

## Usage

1. Navigate to: `https://your-domain.com/pokedex`
2. Enter a Pokemon ID (e.g., `1` for Bulbasaur, `25` for Pikachu)
3. Click **"Szukaj"** (Search)
4. View Pokemon details

## Logging

All Pokemon queries are logged to:
```
var/log/pokemon.log
```

**Log format**:
```
[2024-01-15 10:30:45] main.INFO: Query: 25 - Result: pikachu
[2024-01-15 10:31:12] main.INFO: Query: 1 - Result: bulbasaur
```

## Technical Details

- **Magento Version**: 2.4.x
- **PHP Version**: 8.1+
- **External API**: [PokeAPI v2](https://pokeapi.co/api/v2/)
- **HTTP Client**: GuzzleHTTP (via `Magento\Framework`)
- **Coding Standards**: PSR-2, Magento Coding Standards

## Design Principles Applied

- **SOLID Principles**
  - Single Responsibility: Each class has one reason to change
  - Open/Closed: Open for extension (new API strategies), closed for modification
  - Liskov Substitution: Interfaces can be swapped
  - Interface Segregation: Small, focused interfaces
  - Dependency Inversion: Depend on abstractions, not concretions

- **DRY** (Don't Repeat Yourself)
- **Separation of Concerns**: Controller -> Repository -> Registry -> Service -> DTO -> View
- **Dependency Injection**: No `new` keyword for dependencies
